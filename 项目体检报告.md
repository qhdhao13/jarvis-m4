## 三、可完善之处（按优先级）

### 1. 安全：exec_shell 白名单（进阶说明已提及）

- **现状**：`brain.py` 的 `exec_shell(cmd)` 可执行任意 shell 命令，风险高。
- **建议**：在 system 提示词或工具描述中明确「仅允许以下命令」；或在代码里做白名单（如 `open -a xxx`、`osascript` 调音量等），非白名单一律拒绝并返回提示。  
  详见 **进阶说明.md** 第三节「控制电脑」。

### 2. 启动脚本与 config 一致

- **现状**：`run.sh` 在未设置 `KIMI_API_KEY` 时会检查 Ollama 是否可用；若 `config.yaml` 里 `llm.backend` 为 `kimi` 且已设 Key，则不会检查 Ollama（逻辑正确）。
- **可选**：若希望脚本更智能，可让 `run.sh` 读取 `config.yaml`（例如用 `grep` 或简单 Python 脚本）再决定是否检查 Ollama，避免用户误以为必须起 Ollama。当前行为已可接受。

### 3. 对话历史与记忆

- **现状**：`main.py` 只保留最近 10 条消息，无持久化、无摘要。
- **建议**：按 **进阶说明.md** 第二节，先做「每 N 轮用 LLM 生成简短摘要」再考虑 RAG，避免一次改太多。

### 4. 错误与日志

- **现状**：异常多通过 `print` 和 `traceback.print_exc()` 输出，无统一日志级别、无文件落盘。
- **建议**：使用 `logging` 模块，按级别输出到控制台并可选写入文件（如 `logs/jarvis.log`），便于排查线上/长期运行问题。`tmp/jarvis_run.log` 若曾由某脚本写入，可统一改为写入 `logs/` 并在 `.gitignore` 中忽略。

### 5. 依赖版本与可选依赖

- **现状**：`requirements.txt` 中 `openwakeword` 被注释，与进阶说明一致。
- **建议**：在 `requirements.txt` 顶部或注释中注明「可选依赖：openwakeword（唤醒词）」，并保留 `# openwakeword` 一行，方便复制粘贴安装。

### 6. 双击启动的体验（启动贾维斯.command）

- **现状**：主进程在后台跑，图标在前台；关掉图标窗口后脚本会 `kill $MAIN_PID`，主进程随之结束。
- **说明**：若用户从终端用 Ctrl+C 先结束主进程，图标脚本仍会运行直到关窗，属预期；可在进阶说明或 README 里写一句「关掉图标窗口即退出贾维斯」，减少困惑。

### 7. 测试与 CI（若将来开源或多人协作）

- **建议**：为 `brain` 的工具解析与 `_call_tool` 写少量单元测试（如解析 JSON 工具调用、mock 工具返回）；若需要可加 `pytest` 与 GitHub Actions 跑测试。当前为个人/单机使用可暂不做。
